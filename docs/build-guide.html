Starling is real now. Two birds anywhere on the internet can open a flock, chat over encrypted gossip, and talk over Opus voice — from a terminal, with no company in the middle. This guide walks through the code that exists (Phases 0–3) so you understand every piece you've built, then lays out the road ahead (Phases 4–10) in the same conventions the codebase already uses.

The words we useThe app is Starling. The whole peer-to-peer network is the murmuration. Each node is a bird. A room of birds — one gossip topic, one shared key — is a flock, opened by one bird and joined via its room code. Later, a bird that stays online to keep a community's history is a roost. The metaphor is load-bearing: a real murmuration is thousands of leaderless birds each reacting only to their neighbors, which is exactly how the gossip mesh works.
How to read thisPhases 0–3 are a tour of code that ships today — file by file, with the real source. Phases 4–10 are a roadmap with real code written to drop into the current module layout (event.rs, net.rs, ui.rs, main.rs, friends). Every phase ends with a checkpoint. When a phase touches an existing file, the code is labeled as an edit to that file — never a second competing version.

Phase 0 · Shipped
Foundations

The shape of the whole thing

Starling is one binary with three subcommands — starling setup, starling open, starling join <node-id> — built on two halves that never block each other: a UI loop (ratatui drawing the terminal) and a network task (iroh talking to the murmuration), bridged by two channels.

           ┌─────────────────┐   channel    ┌──────────────────┐
  keyboard →│   UI  (ratatui) │ ───────────► │  Network (iroh)  │→ birds
   screen ←─│   main.rs loop  │ ◄─────────── │  net::run task   │← birds
           └─────────────────┘   channel    └──────────────────┘
              AppEvent enum                    Command enum

src/event.rs — the whole contract, as shipped

/// UI → Network: things the user does.
pub enum Command {
    SendText(String),
    StartCall(EndpointAddr),
    HangUp,
    Quit,
}

/// Network → UI: things that happen in the network.
#[derive(Debug)]
pub enum AppEvent {
    Message(ChatMessage),
    PeerConnected(EndpointId),
    PeerDisconnected(EndpointId),
    Ticket(String),        // our node ID once bound — the invite
    VoiceFrame(Vec<u8>),    // a 20 ms Opus frame from a remote bird
}

/// A chat message that travels over the gossip layer.
#[derive(Clone, Debug, Serialize, Deserialize)]
pub struct ChatMessage {
    pub id: String,      // UUID v4, for dedup
    pub author: String,  // display name from the profile
    pub body: String,
    pub ts: i64,         // unix millis UTC
}

✓ You have: a clean two-task architecture where every future feature is either a new Command arm, a new AppEvent arm, or a new ALPN on the router. Phases 4–10 never break this shape.

Phase 1 · Shipped
Two birds, then any number

Text chat in a flock

One bird runs starling open and gets a room code like BIRD00CCFF. Another runs starling join <node-id> from anywhere on the internet. They're now in the same flock, chatting over gossip.

The trick that makes flocks work: everything derives from the opener

There's no room registry anywhere. The opener's node ID is the room. From it, every bird independently derives the same three things:

src/net.rs — derivation helpers (as shipped)

/// Stable 32-byte TopicId from a name, via SHA-256.
pub fn topic_for(name: &str) -> TopicId {
    let hash = Sha256::digest(name.as_bytes());
    TopicId::from_bytes(hash.into())
}

/// Short room code: "BIRD" + first 6 hex chars of the opener's node ID.
pub fn room_code_from_node_id(node_id: &EndpointId) -> String {
    let bytes = node_id.as_bytes();
    let hex: String = (0..3).map(|i| format!("{:02X}", bytes[i])).collect();
    format!("BIRD{hex}")
}
opener's node ID ──► room code "BIRDA1B2C3"
                      ├──► gossip topic  = topic_for("starling/flock/BIRDA1B2C3")
                      ├──► E2E key       = FlockCrypto::from_room_code(...)   (Phase 3)
                      └──► header color  = the 6 hex chars, read as RGB

That last one is a lovely touch already in ui.rs: room_code_rgb parses the code's hex as a color and draws a swatch in the header — every flock has its own color, for free.

The network loop, as shipped

src/net.rs — the heart of net::run (excerpt)

let endpoint = Endpoint::bind(presets::N0).await?;
endpoint.online().await;
let my_node_id = endpoint.addr().id;

// The opener is bootstrap[0] for joiners, ourselves for openers.
let opener_id = bootstrap.first().copied().unwrap_or(my_node_id);
let room_code = room_code_from_node_id(&opener_id);
let topic = topic_for(&format!("starling/flock/{room_code}"));
let crypto = FlockCrypto::from_room_code(&room_code);      // Phase 3

let _ = evt_tx.send(AppEvent::Ticket(my_node_id.to_string()));

let gossip = Gossip::builder().spawn(endpoint.clone());
let _router = Router::builder(endpoint.clone())
    .accept(GOSSIP_ALPN, gossip.clone())
    .accept(crate::call::VOICE_ALPN, VoiceProto { evt_tx: evt_tx.clone() })
    .spawn();

// `subscribe` (not subscribe_and_join): returns immediately, connects
// bootstrap peers in the background; NeighborUp fires as they land.
let (sender, mut receiver) = gossip.subscribe(topic, bootstrap).await?.split();

loop {
    tokio::select! {
        Some(cmd) = cmd_rx.recv() => match cmd {
            Command::SendText(text) => {
                let msg = ChatMessage { id: uuid::Uuid::new_v4().to_string(),
                    author: name.clone(), body: text,
                    ts: chrono::Utc::now().timestamp_millis() };
                let ciphertext = crypto.encrypt(&postcard::to_stdvec(&msg)?);
                sender.broadcast(ciphertext.into()).await?;
                let _ = evt_tx.send(AppEvent::Message(msg));   // local echo
            }
            /* StartCall / HangUp — Phase 2; Quit breaks */
        },
        Some(event) = receiver.next() => match event {
            Ok(Event::Received(msg)) => {
                if let Some(plain) = crypto.decrypt(&msg.content) {
                    if let Ok(m) = postcard::from_bytes::<ChatMessage>(&plain) {
                        let _ = evt_tx.send(AppEvent::Message(m));
                    }
                }
            }
            Ok(Event::NeighborUp(id))   => { let _ = evt_tx.send(AppEvent::PeerConnected(id)); }
            Ok(Event::NeighborDown(id)) => { let _ = evt_tx.send(AppEvent::PeerDisconnected(id)); }
            Ok(_) => {}
            Err(e) => crate::logger::error(&format!("gossip stream error: {e}")),
        }
    }
}
The one idea that mattersThis tokio::select! loop is the heart of every phase, shipped and future. It waits on "did the user do something" and "did the murmuration do something" simultaneously. Voice added arms to it; video, presence, and roosts will too. The loop shape never changes.

The UI side, as shipped

ui.rs renders two screens off one App struct: a centered name popup when app.name is empty (first launch without a profile), and the chat screen — header (color swatch + room code + truncated invite), message list, a birds panel listing peers with Tab-cycled selection, a status line, and the input box. main.rs drains events into App and maps keys to Commands — dedup on PeerConnected, and selected_peer kept in bounds when birds leave.

Phase 2 · Shipped
Now they can hear each other

Voice

Tab to select a bird, Ctrl+K to call. Your mic streams Opus frames over a direct QUIC connection; theirs plays out your speaker. Ctrl+M mutes.

Voice is the opposite of text: continuous, tiny, worthless if late. So it doesn't touch gossip at all — it rides its own ALPN (starling/voice/0) as QUIC datagrams: unreliable, unordered, no retransmission. A late frame is dropped, which is exactly right.

The pipeline, as shipped

CAPTURE  (voice.rs): mic ─cpal─► f32 ─[mute gate]─► Opus encode ─► mpsc ─► call.rs
TRANSPORT (call.rs): place_call: datagram per 20ms frame ──────────► peer
                     handle_incoming: datagram ─► AppEvent::VoiceFrame
PLAYBACK (playback.rs): Opus decode ─► lock-free ring buffer ─► speaker (cpal)

src/voice.rs — the capture callback (excerpt)

const SAMPLE_RATE: u32 = 48_000;
const FRAME: usize = 960;    // 20 ms @ 48 kHz mono — the Opus standard

let stream = device.build_input_stream(cfg,
    move |data: &[f32], _| {
        acc.extend_from_slice(data);
        while acc.len() >= FRAME {
            let frame: Vec<f32> = acc.drain(..FRAME).collect();
            if muted.load(Ordering::Relaxed) { continue; }   // the mute gate
            let mut out = vec![0u8; 400];
            if let Ok(n) = enc.encode_float(&frame, &mut out) {
                out.truncate(n);
                let _ = net_tx.send(out);      // ~40-byte packet to call.rs
            }
        }
    }, ...)?;
Three shipped details worth noticingMute is an Arc<AtomicBool> shared between the UI thread and the audio callback — no locks on the real-time path, frames are simply discarded while muted. Device selection comes from the profile: start_capture looks up the named device from config::Profile and falls back to the system default. Everything cpal-touching is wrapped in suppress_stderr so ALSA's stderr spam can't tear the raw-mode terminal.

src/call.rs — transport, complete as shipped

pub const VOICE_ALPN: &[u8] = b"starling/voice/0";

pub async fn place_call(endpoint: Endpoint, peer: EndpointAddr,
    mut frame_rx: mpsc::UnboundedReceiver<Vec<u8>>) -> anyhow::Result<()> {
    let conn = endpoint.connect(peer, VOICE_ALPN).await?;
    while let Some(frame) = frame_rx.recv().await {
        let _ = conn.send_datagram(frame.into());
    }   // loop ends when the mic stream drops on hang-up
    Ok(())
}

pub async fn handle_incoming(conn: Connection,
    evt_tx: mpsc::UnboundedSender<AppEvent>) -> anyhow::Result<()> {
    while let Ok(bytes) = conn.read_datagram().await {
        let _ = evt_tx.send(AppEvent::VoiceFrame(bytes.to_vec()));
    }
    Ok(())
}

Hang-up is elegant: Command::HangUp just sets _mic_stream = None in net.rs. Dropping the cpal stream closes the mic channel, which ends place_call's loop, which closes the connection. No teardown protocol needed.

src/playback.rs — the underrun-safe output (excerpt)

// audio thread: pull what's available, pad the rest with silence
let stream = device.build_output_stream(cfg,
    move |data: &mut [f32], _| {
        let n = consumer.pop_slice(data);
        for sample in &mut data[n..] { *sample = 0.0; }
    }, ...)?;

// decode side, called from main's drain loop on AppEvent::VoiceFrame:
pub fn push_opus(&mut self, bytes: &[u8]) {
    let mut pcm = [0f32; FRAME];
    if let Ok(n) = self.decoder.decode_float(bytes, &mut pcm, false) {
        self.producer.push_slice(&pcm[..n]);
    }
}

✓ You have: 1:1 voice between any two birds in the flock — Opus-compressed (~20–40 kbit/s vs ~1.5 Mbit/s raw), encrypted by QUIC TLS 1.3, with mute, device preferences, and WSL2 support. The real-time-thread / ring-buffer pattern here is exactly what video will reuse.

Phase 3 · Shipped
Locks and names

E2E encryption & profiles

Two things landed here: every gossip message is now ChaCha20-Poly1305 encrypted end-to-end, and birds have persistent profiles — a name, audio devices, and a 32-digit code that moves your identity between machines.

Encryption: the flock key

The design is symmetric and room-scoped: every bird derives the same key from the room code, so possession of the invite is membership. Relays and any bird outside the flock see only ciphertext.

src/crypto.rs — complete as shipped

pub struct FlockCrypto { cipher: ChaCha20Poly1305 }

impl FlockCrypto {
    /// All birds in the flock derive the same key from the room code.
    pub fn from_room_code(code: &str) -> Self {
        let key = Sha256::digest(format!("starling/flock/{code}").as_bytes());
        Self { cipher: ChaCha20Poly1305::new(Key::from_slice(&key)) }
    }

    /// Returns [12-byte nonce | ciphertext + 16-byte tag].
    pub fn encrypt(&self, plaintext: &[u8]) -> Vec<u8> {
        let uuid = uuid::Uuid::new_v4();
        let nonce_bytes = &uuid.as_bytes()[..12];
        let ciphertext = self.cipher.encrypt(Nonce::from_slice(nonce_bytes),
            plaintext).unwrap_or_default();
        let mut out = nonce_bytes.to_vec();
        out.extend(ciphertext);
        out
    }

    pub fn decrypt(&self, data: &[u8]) -> Option<Vec<u8>> {
        if data.len() < 12 { return None; }
        self.cipher.decrypt(Nonce::from_slice(&data[..12]), &data[12..]).ok()
    }
}

Integration cost in net.rs was two lines: crypto.encrypt(...) before broadcast, crypto.decrypt(...) before deserializing. Voice needed nothing — a 1:1 QUIC connection is already end-to-end TLS 1.3.

Honest threat model — know what you builtThis is real E2E against relays and outsiders, but the key is derived from the room code, and the code is only 3 bytes of entropy beyond a known prefix (~16.7M possibilities). A motivated attacker can enumerate every possible flock topic and key. Also: one shared key means no forward secrecy and no per-bird identity in the crypto itself. That's a deliberate v0.3 trade — invite-is-membership, simple and honest — and Phase 9 is the planned upgrade (per-bird keys, signed messages, real key exchange). Until then, treat flocks as private-ish rooms, not state-proof channels.

Profiles: a name that survives restarts

config.rs persists a Profile { name, input_device, output_device } as postcard bytes at ~/.config/starling/profile.bin (or %APPDATA%\starling). The clever bit is the 32-digit profile code — your name, packed into 16 bytes, hex-encoded, portable to another machine:

src/config.rs — the portable code (excerpt)

/// 1 byte length + up to 15 bytes of name = 16 bytes = 32 hex digits.
pub fn to_code(&self) -> String {
    let name_bytes = self.name.as_bytes();
    let len = name_bytes.len().min(15) as u8;
    let mut buf = [0u8; 16];
    buf[0] = len;
    buf[1..1 + len as usize].copy_from_slice(&name_bytes[..len as usize]);
    data_encoding::HEXUPPER.encode(&buf)
}
// from_code() reverses this; devices stay machine-local on purpose.

Around it, setup.rs ships a full five-step wizard TUI (starling setup): optional code entry → name → microphone picker → speaker picker → summary + your code. And main.rs degrades gracefully — no profile yet means the in-app name popup, with the wizard available whenever.

✓ You have: encrypted-by-default flocks and a bird identity that persists across launches and machines — with a clear-eyed note about exactly where the crypto stops. This is the "primitive profiles" base that Phase 9 upgrades to signed, cryptographic identity.

/// One RGB frame → terminal lines, two vertical pixels per cell.
pub fn frame_to_lines(img: &RgbImage, cols: u16, rows: u16) -> Vec<Line<'static>> {
    let small = image::imageops::resize(img, cols as u32,
        (rows * 2) as u32, FilterType::Triangle);
    (0..rows).map(|cy| Line::from((0..cols).map(|cx| {
        let (t, b) = (small.get_pixel(cx.into(), (cy*2).into()),
                      small.get_pixel(cx.into(), (cy*2+1).into()));
        Span::styled("▀", Style::new()
            .fg(Color::Rgb(t[0], t[1], t[2])).bg(Color::Rgb(b[0], b[1], b[2])))
    }).collect::<Vec<_>>())).collect()
}

/// ~15fps camera thread → JPEG bytes on a channel (mirrors voice::start_capture;
/// wrap construction in util::suppress_stderr like the audio devices).
pub fn start_camera(tx: mpsc::UnboundedSender<Vec<u8>>)
    -> anyhow::Result<std::thread::JoinHandle<()>> {
    let mut cam = Camera::new(CameraIndex::Index(0),
        RequestedFormat::new::<RgbFormat>(RequestedFormatType::AbsoluteHighestFrameRate))?;
    cam.open_stream()?;
    Ok(std::thread::spawn(move || while let Ok(f) = cam.frame() {
        if let Ok(img) = f.decode_image::<RgbFormat>() {
            let mut jpeg = std::io::Cursor::new(Vec::new());
            if DynamicImage::ImageRgb8(img).write_to(&mut jpeg, ImageFormat::Jpeg).is_ok()
                && tx.send(jpeg.into_inner()).is_err() { break; }
        }
    }))
}

src/call.rs — add video transport beside voice

pub const VIDEO_ALPN: &[u8] = b"starling/video/0";

pub async fn place_video(endpoint: Endpoint, peer: EndpointAddr,
    mut frame_rx: mpsc::UnboundedReceiver<Vec<u8>>) -> anyhow::Result<()> {
    let conn = endpoint.connect(peer, VIDEO_ALPN).await?;
    let mut tx = conn.open_uni().await?;
    while let Some(jpeg) = frame_rx.recv().await {
        tx.write_u32(jpeg.len() as u32).await?;
        tx.write_all(&jpeg).await?;
    }
    Ok(())
}

pub async fn recv_video(conn: Connection,
    evt_tx: mpsc::UnboundedSender<AppEvent>) -> anyhow::Result<()> {
    let mut rx = conn.accept_uni().await?;
    loop {
        let len = rx.read_u32().await? as usize;
        let mut buf = vec![0u8; len];
        rx.read_exact(&mut buf).await?;
        let _ = evt_tx.send(AppEvent::VideoFrame(buf));
    }
}

Phase 5
Who's actually here

Names, presence & stable identity

Goal: the birds panel shows names instead of node-ID stubs, latecomers get recent history, and your room code stops changing every launch.

1 · Stable identity (small change, big payoff)

Today Endpoint::bind(presets::N0) mints a fresh identity each run — so an opener's room code changes every launch. Persist the secret key next to the profile and the same bird keeps the same node ID, the same room code, and (in Phase 9) a signable identity:

src/config.rs — add key persistence

pub fn load_or_create_secret() -> iroh::SecretKey {
    let path = Self::config_dir().join("identity.key");
    if let Ok(bytes) = std::fs::read(&path) {
        if let Ok(arr) = <[u8; 32]>::try_from(bytes.as_slice()) {
            return iroh::SecretKey::from_bytes(&arr);
        }
    }
    let key = iroh::SecretKey::generate(&mut rand::rngs::OsRng);
    let _ = std::fs::create_dir_all(Self::config_dir());
    let _ = std::fs::write(&path, key.to_bytes());
    key
}
// net.rs: Endpoint::builder(presets::N0).secret_key(key).bind().await?

match postcard::from_bytes::<Wire>(&plain) {
    Ok(Wire::Chat(m)) => { let _ = evt_tx.send(AppEvent::Message(m)); }
    Ok(Wire::AboutMe { endpoint_id, name }) => {
        names.insert(endpoint_id, name);
        // re-announce ourselves so THEY learn US (idempotent, deduped by map)
        let _ = evt_tx.send(AppEvent::RosterUpdate(roster_from(&names)));
    }
    Err(_) => {}   // old client or garbage — ignore
}

ui.rs: the birds panel maps each peer through the roster — names.get(id) falling back to id.fmt_short() — so old clients still render. main.rs: one drain arm, AppEvent::RosterUpdate(r) => app.roster = r.

3 · History for latecomers

New ALPN, same pattern as voice: starling/sync/0. Every bird keeps its scrollback in memory already (app.messages); serve it. On join, ask your bootstrap bird for everything recent over a bi-stream: request is (since_ts: i64), response is Vec<ChatMessage>, postcard both ways, capped (say 500 messages). Wire it as a SyncProto registered on the router and a backfill() called after subscribe; results land as AppEvent::HistoryChunk, prepended in main.rs.

Group calls: know the ceilingText gossip scales to thousands of birds. Voice doesn't: everyone-calls-everyone is N² connections — fine to ~4–5 birds meshed, then you need one node forwarding streams (an SFU). That's a natural job for a roost (Phase 7). Don't build it before you have roosts.

✓ Checkpoint: named birds in the panel, history on join, and a room code that's yours for keeps.

Phase 6
More than one conversation

Many flocks at once

Goal: be in several flocks simultaneously — a rail of flocks on the left, Alt+↑/↓ to switch, each with its own topic, key, and scrollback.

Everything a flock needs already derives from one room code. So "many flocks" is a map keyed by room code, holding what today are singletons:

Phase 7
A place to come back to

Roosts: servers & channels

Goal: a community whose history, channels, and membership survive everyone going offline — because one bird, the roost, stays.

A roost is the same binary in --roost mode: a bird that stays online, persists everything to disk, and is authoritative for its community. A community is a set of channels; each channel is just a flock topic derived from the roost's ID plus the channel name — all the Phase 1–6 machinery reused.

roost node ID ──► community code (same BIRDxxxxxx scheme)
                   ├── channel "#general"  = topic_for("starling/roost/{code}/general")
                   ├── channel "#builds"   = topic_for("starling/roost/{code}/builds")
                   └── roost subscribes to ALL of them, and persists

src/roost/store.rs — new: sled-backed history

cargo add sled

pub struct Store { db: sled::Db }
impl Store {
    /// key = "{channel}/{ts:020}" → range scans give time order for free
    pub fn append(&self, chan: &str, m: &ChatMessage) -> anyhow::Result<()> {
        self.db.insert(format!("{chan}/{:020}", m.ts).as_bytes(),
            postcard::to_stdvec(m)?)?;
        Ok(())
    }
    pub fn since(&self, chan: &str, ts: i64) -> Vec<ChatMessage> {
        self.db.range(format!("{chan}/{:020}", ts).as_bytes()..
                      format!("{chan}/~").as_bytes())
            .filter_map(|kv| postcard::from_bytes(&kv.ok()?.1).ok()).collect()
    }
}

The roost's main loop is net::run minus the TUI: subscribe to every channel topic, store.append each decrypted message, and serve Phase 5's starling/sync/0 from disk instead of memory — so backfill now survives total flock silence. Channel metadata (the list, names, order) lives in a signed RoostState blob the roost gossips on a control topic; clients render it as the channel rail from Phase 6's UI.

Phase 8
Who can do what

Roles & permissions

Goal: owners and mods manage the roost; members can't escalate; the ban hammer is enforced where it counts.

Copy Discord's proven model: permissions are a bitmask, roles bundle permissions, a member's effective perms are the OR of their roles. And the iron rule: the roost enforces, the client only decorates. Hiding a key in the TUI is courtesy; a modified client can send anything, so every privileged action is checked by the roost before it's applied.

src/roost/perms.rs — new

cargo add bitflags

bitflags! {
    #[derive(Clone, Copy, Serialize, Deserialize)]
    pub struct Perm: u64 {
        const VIEW_CHANNEL = 1<<0;  const SEND_MESSAGE = 1<<1;
        const MANAGE_MSGS  = 1<<2;  const KICK         = 1<<3;
        const BAN          = 1<<4;  const MANAGE_CHANS = 1<<5;
        const MANAGE_ROLES = 1<<6;  const ADMIN        = 1<<7;
    }
}

pub struct Role { pub name: String, pub color: (u8,u8,u8),
                  pub perms: Perm, pub position: u16 }

// enforcement, in the roost's command handler:
fn handle_ban(&mut self, from: &EndpointId, target: &EndpointId) -> Result<()> {
    anyhow::ensure!(self.effective(from).contains(Perm::BAN), "not allowed");
    anyhow::ensure!(self.outranks(from, target), "can't ban equal/higher");
    self.bans.insert(*target);
    self.broadcast_state();
    Ok(())
}

Client side: moderation Commands (Ban, Kick, DeleteMessage) sent to the roost over a control stream; ui.rs colors names by top role (the birds panel already renders per-item styles) and gates the mod menu on app.my_perms; main.rs binds the keys only when permitted. Rejections come back as a status-line notice.

✓ Checkpoint: roles colored in the panel, enforced at the roost, impossible to escalate from a client.

Phase 9
Closing the gap from Phase 3

Real cryptographic identity

Goal: fix everything the Phase 3 callout warned about — per-bird keys, signed messages, high-entropy invites, forward-secret DMs, and profiles that can't be forged.

  1. Sign every message. With Phase 5's persistent SecretKey, your EndpointId is an ed25519 public key. Extend the Wire envelope to { payload, author_pk, sig } — receivers verify before rendering, and author names become unforgeable. (Today, any flock member can claim any name.)
  2. High-entropy invites. Keep deriving the display code from the node ID, but derive the key from a random 32-byte secret carried in the invite: starling open prints <node-id>.<base32-secret>. Brute-forcing 16.7M room codes stops working; the pretty BIRDxxxxxx stays as the label and color.
  3. Sealed 1:1 DMs. cargo add crypto_box — encrypt directly to a bird's public key for whisper messages inside a flock (/msg <bird>). The flock relays it; only the recipient reads it.
  4. Group E2E with real membership. For roost channels that want E2E despite the roost (private channels), adopt MLS via openmls — designed exactly for encrypting to a changing member set, with forward secrecy. This is the largest item; do it last and only where it's needed.
  5. Signed profiles. Extend config::Profile with pronouns, status, accent color, and a small avatar (rendered by Phase 4's frame_to_lines in a popup) — the whole struct signed by the identity key, gossiped as Wire::Profile, verified on receipt, cached per bird. The 32-digit code grows into a signed export.
Sequencing mattersItems 1–2 are small and remove the two real weaknesses (name spoofing, low-entropy keys) — do them together as one release. Item 3 is a weekend. Items 4–5 are features, not fixes; let demand pull them.

✓ Checkpoint: unforgeable authorship, unguessable flocks, private whispers — the Phase 3 threat-model callout deleted because it no longer applies.

Phase 10
The last mile of independence

Own the murmuration

Goal: no single company anywhere in the flight path — plus the long tail of features that make it feel finished.

Since Phase 1, connectivity has ridden on n0's public relays and discovery. That's the one centralized dependency left. iroh makes each piece swappable:

  • Self-hosted relays. Run iroh-relay and point endpoints at it via RelayMode::Custom. A roost is the natural host — a community that runs its own roost can run its own relay beside it. E2E crypto means relays forward ciphertext they can't read.
  • Company-free discovery. Swap n0's DNS discovery for pkarr (public-key records on the BitTorrent mainline DHT) so resolving a bird's address depends on no one in particular.
  • Invites as links. starling://join/<node-id>.<secret> URL scheme, registered by the installer.

Zooming out

The through-line

Look at what never changed from the first commit to the Phase 10 plan: the UI-loop / network-task split, the select! loop with two match arms, one room code deriving everything a flock needs, and a new ALPN for every point-to-point job (voice, video, sync, control). Voice fit that shape. E2E fit it in two lines. Video, roosts, and permissions will fit it too. You didn't just build features — you built a shape that features fall into.

Suggested rhythm from here: Phase 4 (video) is self-contained fun. Phase 5 is the highest-leverage small work — stable identity and names make everything after it nicer. Phase 9 items 1–2 are worth pulling forward the moment strangers start sharing room codes. Phases 7–8 are the real commitment; don't start a roost until flocks feel boring.

No company owns the murmuration. One bird opens a flock, another joins, and the sky fills in between — built in the open, one runnable phase at a time.
